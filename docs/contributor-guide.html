üîÑ Contribution Workflow

The Standard Process

  1. Fork the repository (only needed once)
  2. Create a topic branch for your changes
  3. Make atomic commits with clear messages
  4. Push to your fork and create a pull request
  5. Address review feedback through discussion and updates
  6. Squash commits if requested by reviewers

Repository Selection

Which repository to use:

  • GUI-only changes: https://github.com/bitcoin-core/gui
  • Everything else: https://github.com/bitcoin/bitcoin
  • Global refactoring: Main repository even if it touches GUI

Commit Guidelines

# Commit message format
component: Short description (50 chars max)

Detailed explanation of what and why, not how.
Can span multiple lines and include technical details.

# Examples:
consensus: Add new opcode for BIP-XXXX OP_CHECKAWESOMESIG
net: Automatically create onion service, listen on Tor
qt: Add fee bump button
log: Fix typo in log message

üì§ Pull Request Guidelines

PR Title Prefixes

Prefix Usage Example
consensus Consensus critical code consensus: Add new opcode validation
net / p2p Peer-to-peer network code net: Improve peer discovery
rpc / rest RPC and REST APIs rpc: Add new wallet endpoint
wallet Wallet functionality wallet: Improve coin selection
qt / gui GUI changes qt: Add dark mode support
test / qa Tests and QA test: Add functional test for feature
build Build system build: Update CMake requirements
doc Documentation doc: Update developer notes
refactor Code refactoring refactor: Extract common validation logic

PR Description Requirements

Your PR description should include:

  • What: Clear description of the change
  • Why: Justification and motivation
  • How: Technical approach (if not obvious)
  • Testing: How you tested the change
  • References: Related issues, discussions, BIPs
‚ö†Ô∏è Avoid @mentions: Don't include @username mentions in PR descriptions as they'll be preserved in commit messages and notify users every time the code is copied.

üîç Review Process

Types of Review

Conceptual Review

  • Concept ACK: "I agree with the general goal"
  • Concept NACK: "I disagree with this approach" (requires rationale)
  • Approach ACK: "I agree with the implementation approach"
  • Approach NACK: "Different approach needed" (with reasoning)

Code Review

  • ACK [commit-hash]: "I have reviewed and tested the code"
  • NACK: "I object to this change" (with detailed reasoning)
  • Tested ACK: "I have tested the functionality manually"
  • Code review ACK: "Code looks good but I haven't tested"

Review Standards

All PRs must:

  • Have a clear use case or fix a demonstrable bug
  • Be well peer-reviewed
  • Include appropriate tests
  • Follow code style guidelines
  • Not break existing test suite
  • Update relevant documentation

Consensus Changes

üö® Special Requirements for Consensus Changes:
  • Must be preceded by extensive mailing list discussion
  • Require a numbered BIP (Bitcoin Improvement Proposal)
  • Need widespread technical consensus
  • Higher review standards and longer timeline
  • Consider using Bitcoin Inquisition for testing first

Finding Reviewers

# Use git blame to find relevant experts
git blame src/wallet/wallet.cpp

# Check recent commits in related areas
git log --oneline --author="maintainer-name" src/net/

# Join IRC and politely ask for review
# Wait at least a week before following up

üß™ Testing & Quality Assurance

Test Types

Test Type Command Purpose
Unit Tests ./bin/test_bitcoin Test individual functions/classes
Functional Tests test/functional/test_runner.py End-to-end integration testing
Fuzz Tests ./bin/test_bitcoin -fuzz Find crashes and undefined behavior
Util Tests test/util/test_runner.py Test utility scripts

Test Writing Guidelines

# Unit test example location
src/test/wallet_tests.cpp

# Functional test example location
test/functional/wallet_basic.py

# Test naming convention
test_feature_description.py
test_rpc_command_validation.py

# Running specific tests
./bin/test_bitcoin --log_level=all --run_test=wallet_tests
test/functional/wallet_basic.py

Manual QA Testing

Testing Checklist:

  • Test on multiple operating systems if possible
  • Verify backward compatibility
  • Test edge cases and error conditions
  • Validate RPC interface changes
  • Check for performance regressions
  • Test with different configuration options

üéì Advanced Topics

Release Process

Testing Release Candidates:

  • Major releases happen every 6-8 months
  • Release candidates need thorough testing
  • Special testing guides are provided for each RC
  • Testing RCs is a valuable contribution opportunity

Security Considerations

üîí Security Guidelines:

  • Never test with real Bitcoin on mainnet
  • Use regtest or signet for testing
  • Report security issues privately via security@bitcoincore.org
  • Follow responsible disclosure for vulnerabilities
  • Be cautious with consensus changes

üéØ Best Practices Summary

üí° Key Success Factors:

  • Start small: Begin with reviews and small fixes
  • Communicate early: Discuss ideas before implementing
  • Be patient: Quality review takes time
  • Stay humble: Learn from feedback
  • Test thoroughly: Quality over speed
  • Join the community: Participate in IRC and discussions
  • Read the code: Understanding the codebase is crucial
  • Follow conventions: Consistency matters

Communication Etiquette

ü§ù Professional Interaction:

  • Be respectful and constructive in all communications
  • Many contributors are volunteers - be patient
  • Give people 3-7 days to respond before following up
  • Focus on technical merit, not personal preferences
  • Acknowledge and thank reviewers for their time
  • Be open to criticism and alternative approaches

Time Investment Expectations

‚è∞ Realistic Timeline Expectations:

  • Small fixes: 1-4 weeks from PR to merge
  • Medium features: 1-6 months
  • Large features: 6+ months
  • Consensus changes: 1+ years
  • Learning curve: 3-6 months to become effective

üéâ Ready to Contribute?

Here's what to do next:

  1. Set up your development environment
  2. Join #bitcoin-core-dev on IRC
  3. Attend a PR Review Club meeting
  4. Find a "good first issue" to work on
  5. Start reviewing other contributors' PRs

Remember: The Bitcoin Core community values quality, security, and consensus. Take your time, learn continuously, and contribute thoughtfully!
